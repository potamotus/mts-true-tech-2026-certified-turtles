## Задача (из task.md — Трек 1: GPTHub)

### Проблематика
Сотрудники тратят время на переключение между несколькими ИИ-сервисами (текст, картинки, аудио, анализ файлов, поиск). Нужен единый интерфейс-мессенджер, который сам подбирает нужный инструмент/модель под задачу и безопасно хранит контекст.

### Полное описание
Разработать рабочее ИИ-пространство с единым чат-интерфейсом. Через одно окно: текстовые запросы, голосовые сценарии, генерация изображений, работа с аудио, анализ изображений, ответы по файлам, вызов инструментов (поиск в интернете, веб-парсинг).

**Ключевая идея:** не набор кнопок вокруг LLM, а понятный и надёжный UX. Система должна **определять тип задачи и автоматически выбирать подходящую модель или инструмент**: VLM — для изображения, ASR — для аудио, LLM — для текста, image generation — для иллюстраций. Пользователю доступен и **ручной режим** выбора модели.

**Важно:** система должна использовать ранее сохранённый контекст и факты о пользователе для последовательного диалога на длинной дистанции.

### Обязательные сценарии
1. Единый чат-интерфейс (OpenWebUI как база)
2. **Автоматический роутинг моделей** — система сама определяет тип задачи и выбирает модель
3. **Ручной выбор модели** — пользователь может переопределить
4. **Мультимодальность** — текст, голос, изображения, файлы
5. **Долгосрочная память** — сохранение контекста и фактов о пользователе
6. Голосовой ввод → текстовый ответ
7. Прикрепление файла → анализ
8. Ссылка на веб-страницу → парсинг + ответ

### Доп. сценарии (бонус)
- Deep research
- Генерация презентаций
- Расширенные tool-calls

### Технические ограничения
- База — **OpenWebUI** или его форк с сохранением чатовой логики
- Стек: Python и/или JavaScript/TypeScript
- Запросы к моделям — через **MWS GPT**

## Задача для анализа

**Сделай полный анализ и определи способы создания системы роутинга (автоматического выбора модели) для GPTHub.**

### Что нужно сделать

#### 1. Анализ сценариев и требований
- Изучи задачу GPTHub (единый чат, мультимодальность, авто/ручной выбор модели, память)
- Для каждого из 12 сценариев определи:
  - Тип модальности (text/speech/vision/image/embeddings)
  - Требования к латентности (real-time vs async)
  - Требования к качеству (простой ответ vs флагманский)
  - Ограничения по контексту (длина промпта, история)
  - Кандидат-модели из каталога MWS GPT (основная + fallback)

#### 2. Карта моделей -> задач
- Составь матрицу: сценарий -> основная модель -> fallback модель -> почему
- Учти специализацию внутри модальности:
  - **Текст:** general (mws-gpt-alpha, qwen3-32b) vs reasoning (deepseek-r1, QwQ) vs lightweight (llama-8b) vs flagship (Qwen3-235B, glm-4.6)
  - **Vision:** стандартная (qwen2.5-vl) vs мощная (qwen2.5-vl-72b) vs эффективная (qwen3-vl-30b)
  - **Speech:** качество (whisper-medium) vs скорость (whisper-turbo-local)
  - **Image:** качество (qwen-image) vs скорость (qwen-image-lightning)
  - **Embeddings:** bge-m3 vs qwen3-embedding-8b

#### 3. Архитектура роутинга — сравни подходы

Предложи и детально разбери **минимум 4 подхода**:

**A. Static Rule Router**
- Жёсткие правила: есть картинка → VLM, есть аудио → Whisper, есть код → coder
- Плюсы/минусы, схема, latency, когда подходит

**B. Intent Classifier Router**
- Лёгкий классификатор (или маленькая LLM) определяет интент запроса
- Роутит на специализированную модель
- Плюсы/минусы, схема, latency

**C. LLM-as-Router (Meta-модель)**
- LLM сама решает какую модель использовать (function calling / structured output)
- Плюсы/минусы, overhead, когда оправдано

**D. Hybrid Router (рекомендуемый для хакатона)**
- Модальность определяется по типу входных данных (правила)
- Внутри текстовых задач — intent classification или LLM-роутинг
- Fallback-цепочка для отказоустойчивости
- Плюсы/минусы, схема

Для каждого подхода:
- Схема компонентов (ASCII-диаграмма)
- Поток данных: запрос → роутер → модель → ответ
- Обработка ошибок и fallback
- Оценка latency (быстро/средне/медленно)
- Сложность реализации (низкая/средняя/высокая)

#### 4. Внутри-модельный роутинг (parameter routing)
Помимо выбора модели, продумай настройку параметров **внутри** одной модели:
- **System prompt routing** — разные system-промпты для разных задач на одну модель
- **Parameter routing** — temperature, max_tokens, top_p разные для: чата, кода, перевода, creative-задач
- **Context window management** — как обрезать/сжимать историю для разных задач
- **Streaming vs non-streaming** — когда стримить, когда ждать полный ответ

#### 5. Интеграция с OpenWebUI — как встроить роутинг

Изучи архитектуру OpenWebUI и определи:

**Вариант A — OpenWebUI Functions/Pipelines**
- Кастомная функция-перехватчик перед отправкой запроса
- Анализирует запрос, подменяет модель
- Плюсы: нативная интеграция, минимум кода
- Минусы: ограниченная гибкость

**Вариант B — Прокси-сервер (middleware)**
- Свой FastAPI-прокси между OpenWebUI и MWS GPT API
- Полный контроль над роутингом
- Плюсы: максимальная гибкость
- Минусы: ещё один сервис

**Вариант C — Форк OpenWebUI**
- Прямая модификация кода OpenWebUI
- Плюсы: полный контроль
- Минусы: сложно поддерживать, конфликты с апдейтами

**Вариант D — Комбинированный**
- OpenWebUI Functions для простого роутинга + прокси для сложного
- Плюсы/минусы

Для каждого:
- Какие файлы OpenWebUI нужно тронуть
- Сколько кода примерно нужно
- Совместимость с обновлениями

#### 6. Система памяти (долгосрочный контекст)

Роутинг должен учитывать память. Проанализируй:

- **Хранение фактов** — где и как хранить информацию о пользователе (векторная БД, ключ-значение, граф)
- **Извлечение контекста** — как перед запросом к LLM подтягивать релевантные факты
- **Обновление памяти** — когда и как обновлять хранилище фактов (после каждого диалога, по триггерам)
- **Роутинг с учётом памяти** — если в памяти есть контекст, это влияет на выбор модели?
- **Эмбеддинги для поиска** — bge-m3 или qwen3-embedding-8b для семантического поиска по памяти

#### 7. Fallback и отказоустойчивость

- Каскад fallback для каждого сценария (flagship → mid → lightweight)
- Что делать при таймауте / rate limit / ошибке модели
- Кэширование повторяющихся запросов
- Graceful degradation: если VLM упала — сказать пользователю, а не молчать
- Retry-логика с exponential backoff

#### 8. Рекомендация

Выбери **один лучший подход** для GPTHub и обоснуй:

- Почему именно этот подход (хакатон, 3 дня, команда 4-5 человек, OpenWebUI как база)
- Конкретная схема реализации: какие файлы, какие функции, какой поток
- Оценка сложности (низкая/средняя/высокая)
- Что делаем в первую очередь (MVP роутинга), что — во вторую (полировка)
- Какие сценарии реалистичны за 3 дня, какие — нет (будь честен)
